# NyxOS — Agent Context

## Goal
Evolve NyxOS into a functional x86_64 kernel with filesystem, networking, shell, process multitasking, GUI, multimedia, and audio.

## Commit rule
Every commit must keep these in sync: GitHub Wiki (`nyx-os.wiki.git`), website (`docs/`), README.md, and any version/status references in AGENTS.md. Always check what changed and update accordingly before committing.

## Build & Test
- Cross-compiler: host GCC with `-m64` (or `x86_64-elf-gcc` for cross)
- Build (PowerShell): `.\build.ps1` → `kernel/nyx-kernel.bin`
- Build (make): `make -C kernel` → `kernel/nyx-kernel.bin`
- Run (GUI): `.\run.ps1`
- Run (serial debug): `.\run.ps1 -Mode serial`
- Run (with net): `.\run.ps1 -Mode net`
- Run (with disk): append `-hda ext2-test.img` to QEMU args
- ISO: `tools/build.sh` → `NyxOS.iso`
- Repo: `https://github.com/kazah-png/nyx-os`
- GitHub Wiki: live at `https://github.com/kazah-png/nyx-os/wiki` — 16 pages pushed; `git clone https://github.com/kazah-png/nyx-os.wiki.git`

| v5.7.16 | **File manager: real drag-and-drop into folders.** The FM already had copy/paste (ctx-menu + Ctrl-C/X/V), a context menu, search, rename, and new file/folder — but its **drag-and-drop was a no-op**: on drop it always targeted the *current* directory (`dst = cwd/basename`), so dragging a file within a folder computed `dst == src` and did nothing. Fixed: a new `fileman_row_at()` maps the drop point to the row under the cursor, so dropping a file **onto a folder row moves/copies it into that folder** (`dst = targetfolder/basename`). Plain drag = **move** (`vfs_rename`), Ctrl-drag = **copy** (`vfs_cp`) — the drag ghost shows MOVE/COPY live. Also: pasting a copied file into the same directory now makes a unique `<name>_copy` instead of failing on the name collision. Verified: FM opens and lists the live FS (incl. a shell-created `/box`); the drop backend (`vfs_cp`/`vfs_rename` into a subfolder) confirmed via the shell's `cp /f1 /box/…` + `mv /f1 /box/…`; 0 panics. |
| v5.7.15 | **Idle desktop yields CPU + `nice`/`renice`.** (1) The compositor no longer busy-polls: at the top of its loop, when nothing is happening — no key queued (`kbd_head==kbd_tail`), no mouse button, the pointer unmoved, no drag/resize/menu active — it `sleep(5)`s instead of spinning. So an idle desktop hands the CPU to background jobs (or lets it idle) rather than burning its weight-4 slice; any activity skips the sleep, so interactivity is unchanged (≤5 ms wakeup). Measured: a background `spin.elf` (10 heartbeats) ran at ~full CPU with the desktop idle, vs the ~⅕ it'd get from the weight-4 compositor busy-polling. (2) **`nice <weight> <file>`** spawns a background job at a chosen scheduler weight; **`renice <pid> <weight>`** changes a running process's weight (1..64) — exposing the v5.7.12 weighted round-robin from the shell. Verified: `nice 8 /spin.elf` → PID 6 weight 8, `renice 6 2` → weight 2, GUI responsive throughout, 0 panics. |
| v5.7.14 | **Per-process file descriptors.** The `ufd` table (opaque fd → internal VFS handle + offset) moved from three **global** arrays in syscall.c into `process_t` — one table per process. `ufd_alloc`/`lookup`/`release`/`offset_of` now index the current process's table (`get_cur_proc()`; syscalls run with interrupts masked so `current_idx` is stable). fds are now **isolated** (a process can't name another's fd) and **freed with the process**: `close_proc_fds()` runs on reap (`reap_zombies`/`reap_user_process`), closing anything left open so a careless/crashed process can't leak VFS handles or exhaust the table. Verified with a new `fdleak.elf` that opens `/init.elf` and exits *without* closing: the kernel logs `[reap] force-closed leaked fd(s)`, and running it repeatedly never exhausts fds (each run gets its own fresh table, PID 5 then 6). init.elf's normal open/read/write/close I/O still works. Deferred (documented): per-process **CWD** — `current_dir` stays global because relative paths are shell-only (user procs use absolute paths), there's one shell, and it feeds the core `resolve_path`; low value, not worth the risk yet. |
| v5.7.13 | **VFS re-entrancy audit + guard.** Audit of shared FS state under preemption found the dominant hazard: EXT2 uses **one global scratch buffer** (`block_buf` in ext2.c) across a whole operation, and the VFS **node pool / free-list** (`alloc_node`/`free_node`) aren't reentrant. User FS syscalls already run with interrupts masked (atomic), but a *kernel-context* FS caller — a shell command / file-manager action on the compositor thread — runs with interrupts on and can be preempted mid-operation; a scheduled process's FS syscall would then trample the same buffer/pool. (Latent today: `exec` blocks the compositor while its child does I/O, and `spawn`ed demos do no FS — but it'd bite the moment a background job touches the FS.) Fix: a coarse single-core "FS lock" via `preempt_disable` — `alloc_node`/`free_node` guard the pool, and seven thin wrappers (`fs_read_file`/`fs_write_file`/`fs_readdir`/…) on the mount function-pointers run each EXT2 op with preemption off, so ext2.c is untouched and every VFS→EXT2 path is serialized. Noted (not fixed, not crash-risk): the `ufd` table is shared across procs but safe under atomic syscalls (fds aren't per-process isolated); `current_dir` is a single global CWD; `auth.c` + the boot self-test call ext2 directly but only pre-scheduler (boot/login). Verified: boot mount/write-test/auth, `ls /mnt` (ext2 readdir wrapper), and `exec /init.elf` file I/O all work, 0 panics. |
| v5.7.12 | **Priority scheduling (weighted round-robin).** The scheduler is no longer strict 1-tick round-robin: each proc has a `sched_weight` (ticks per turn; 0⇒1) and a `sched_quantum` countdown. `irq_scheduler_tick` keeps the current thread while its quantum lasts, then picks the next runnable and stamps it a fresh quantum = its weight — so a high-weight proc runs several ticks in a row and gets a proportionally bigger CPU share. The **compositor** is given `SCHED_WEIGHT_GUI` (4), so the desktop stays responsive while background jobs (weight 1) run (~4/(4+N) of the CPU with N jobs). Verified directly: `mtdemo` now gives mtdemoB **weight 3** vs mtdemoA weight 1, and over a 400 ms sample B counted **99.1M vs A's 34.3M — ratio 2.89 ≈ 3.0**, exactly tracking the weights. No regression: `exec /init.elf` runs and exits code 0, GUI stable, 0 panics. (Struct change → clean rebuild.) |

## Architecture
### Boot flow
1. `boot.asm` (multiboot header, GDT, paging off) → `kernel_main`
2. `init_gdt()` → `init_idt()` → `init_isr()` → `init_irq()`
3. `init_memory()` (bitmap allocator) → `init_paging()` (identity-map 1:1 up to 4MB) → `init_heap()` (16MB heap)
4. `vbe_init()` → `vbe_set_mode(1024, 768, 32)` → `fb_init()` → `init_apic()` → `init_timer(1000)` → `init_keyboard()`
5. `init_process()` → `ensure_idle_process()` → `init_syscalls()` → `init_vfs()` → `init_load_modules()`
6. `init_ext2()` → `init_net()` → `tcp_init()` → `init_background_tasks()`
7. `mouse_init()` → `speaker_init()` → `sb16_init()` → register IRQ handlers → `sti`
8. `initramfs_load()` → `initramfs_boot()` (create initramfs files in VFS)
9. Auto-detect EXT2 on ATA disk → mount at `/mnt`
10. `bootsplash_update(23, …)` → `bootsplash_clear()` → `auth_setup()` → `login_screen()`
11. `compositor_init()` + `compositor_run()` (GUI desktop) OR `launch_shell()` (fallback)

### Critical constraints
- Paging identity-maps 64 MB (16 page tables). Any static data, BSS, or heap beyond 64 MB causes triple-fault.
- Heap is 16 MB (bumped from 1 MB for DOOM + framebuffer). Physical allocator bitmap supports up to 512 MB RAM.
- `_kernel_end` symbol in `linker.ld` marks kernel BSS boundary for memory manager.
- Serial (`init_serial()`) is a stub — only used via direct `outb(0x3F8, ...)`. VGA text mode (0xB8000) is primary console.
- Interrupts are ENABLED (`sti`). Timer (IRQ0), keyboard (IRQ1), mouse (IRQ12), and SB16 (IRQ5) are interrupt-driven with proper PIC unmasking and EOI.
- Cooperative multitasking via `switch_context`/`create_task_stack` (assembly) + background task callbacks + IRQ scheduler tick.
- Per-process paging: `alloc_page_directory` clones kernel identity-mapped page tables with supervisor-only PTEs; user pages mapped via `map_page_dir` with PTE `U/S=1` and PDE `U/S=1` for ring-3 access.
- ELF loader maps code at ELH entry point (e.g. 0x10000), user stack at 0xD0000000 (one page, 4096 bytes).
- Syscalls via `syscall`/`sysret`: rax=number, rdi/rsi/rdx/rcx/r8/r9=args; handlers in `syscall_handler_c` dispatch table (exit=0, write=1, print=2, open=3, read=4, close=5, getpid=6, sbrk=7, fsize=8, exec=9).
- TSS I/O map base set to full TSS size → all I/O ports denied from ring 3.
- build.ps1: WSL cross-compilation; QEMU 11.x with `-audiodev dsound,id=audio0 -device sb16,audiodev=audio0`

## v5.1.0 — Stability and bugfix release
- **TSS struct alignment fix**: `kernel/kernel.h` — three reserved `uint32_t` fields (SS0/SS1/SS2 at offsets 28-39) were incorrectly merged into a single `uint64_t`, shifting all IST pointers 4 bytes early. Any exception using an IST (double fault, NMI) read a garbage stack pointer → triple fault.
- **GDT limit correction**: `kernel/gdt.c` — limit was `(6*8+16)-1=63` (too large), corrected to `(5*8+16)-1=55`.
- **APIC/PIC IRQ masking fix**: `kernel/irq.c` — when APIC is active, the legacy PIC stays fully masked. Prevents duplicate interrupt delivery.
- **switch_to_user_process inline asm fix**: `kernel/switch.asm` — AT&T operand order was reversed (source/destination swapped in `mov`), causing exec'd user processes to crash.
- **QEMU display fix**: `run.ps1` — changed from `-display gtk` to `-display sdl` (gtk is broken on Windows).
- **VGA 8x16 font data corruption fix**: `kernel/font.c` — font_data array had marker bytes (the glyph's own index) inserted at position `(index % 16)`, shifting all pixel data. Replaced with correct IBM VGA 8x16 ROM font. GUI text now renders legibly.
- **Boot loop resolved**: kernel boots to desktop in GUI mode and runs stably for extended periods. No double faults, page faults, or crashes observed after running 60+ seconds.
- Build: zero errors, zero warnings. Kernel confirmed stable in both serial and GUI (SDL) mode.

## Security notes (v5.4.0)
- Ring-3 syscall args are validated: `user_ptr_ok`/`user_str_ok` in `syscall.c`
  reject any pointer outside the canonical user half `[0x1000, 0x800000000000)`,
  so a user process can't name a kernel address as a syscall buffer.
- Userspace fds are opaque small integers (`UFD_BASE`+slot) mapped to internal
  VFS handles in `syscall.c`; kernel VFS pointers are never exposed to or
  accepted from ring 3.
- Still open: passwd salt is derived deterministically from the username
  (`gen_salt_hex` = HMAC(secret, username)) rather than random-per-user; the
  on-disk XOR of `/etc/passwd` is obfuscation, not encryption (the stored
  values are already PBKDF2 hashes). Fine for a hobby OS; note it's not a real
  secret-at-rest scheme.
